You are an autonomous research agent working on **Lattice Reduction Benchmarking for LWE Concrete Security Estimation**.

## Goal
The security of NIST's post-quantum cryptographic standards (Kyber/ML-KEM, Dilithium/ML-DSA) depends on the hardness of LWE. NIST claims these provide 128/192/256-bit security, but these claims rest on theoretical cost models that may not match what lattice reduction actually achieves in practice. Crucially, the security estimate changes depending on whether the attacker is classical (Core-SVP: 2^{0.292·β}) or quantum (Grover-accelerated sieving: 2^{0.265·β}).

Your job has two parts:

**Part A — Cost Model (what PRED_SCORE measures):** Fit an empirical model that predicts BKZ wall-clock runtime as a function of (n, q, σ, β, attack_type). This model predicts **log₂(seconds)** — machine-specific, not security bits. Improving PRED_SCORE means your model better predicts how long lattice attacks take.

**Part B — Security Assessment (heuristic, from raw δ data):** From the empirical root-Hermite factor δ(β) curves in the benchmark data, extrapolate the β* needed to break NIST-scale instances. Then convert: classical bits = 0.292 × β*, quantum bits = 0.265 × β*. This gives a heuristic over/under-conservative verdict for each NIST scheme. Part B runs automatically but is NOT what the agent optimizes — it uses raw benchmark data, not the fitted model.

The agent's loop optimizes Part A. Part B is a separate downstream analysis that uses raw benchmark δ data from `lattice_benchmark.pt` — it does NOT use the fitted model, so improving PRED_SCORE does not directly change the security verdicts unless you also add δ-prediction to the model (see Research Direction 7).

## Scope and Limitations
This benchmark implements **two attacks**:
- **Primal uSVP** (Kannan embedding → BKZ)
- **Dual distinguishing** (q-ary dual lattice → BKZ → inner-product test)

**Not implemented**: hybrid meet-in-the-middle, Arora–Ge. These are noted as future work.

**NIST extrapolation caveat**: Kyber is Module-LWE, not plain LWE. Treating it as plain LWE with n = k×256 gives a rough proxy, not a precise security estimate. A proper analysis would use the lattice-estimator tool with module-aware attacks.

Other known limitations:
- `svp_calls_est` is a rough count, not measured from the solver.
- No comparison against the lattice-estimator tool (only a homegrown Core-SVP proxy).
- No progressive BKZ strategies (pump-then-jump, dimensions-for-free).
- GSA formula used for β estimation is a simplified leading-term proxy.
- Theory predictions for dual rows use the primal model (no proper dual theory).

**PRED_SCORE** = 0.3 × R²_interp + 0.5 × R²_extrap + 0.2 × theory_rank_correlation.

## Data (generated by prepare.py)
`lattice_benchmark.pt` contains:
- LWE instances at n ∈ {40, 50, 60} (training) and n ∈ {70, 80} (extrapolation)
- Secrets are **small** (CBD/Gaussian, same distribution as errors) — matching Kyber
- Two modulus choices: q ≈ n^{1.5} ("small") and q ≈ n² ("large")
- Three error distributions: Gaussian σ=1.0, Gaussian σ=3.19, CBD(2)
- Both primal and dual attacks at BKZ block sizes β ∈ {10, 15, 20, 25, 30, 35}
- For each: wall-clock time, root-Hermite factor δ, attack success
  - Primal success: **exact** mod-q check against true (e, -s, 1)
  - Dual success: **heuristic** (normalized score < 3.0); quantitative stats stored
- Core-SVP theoretical predictions (primal uSVP model only)
- Theory rank correlation in PRED_SCORE uses **primal extrapolation rows only**
- Train/test split is by **instance group** (all β values for one LWE instance stay together)

## Research Directions
1. **Feature Engineering:** Baseline uses 10 features (log₂(n), log₂(q), log₂(σ), β, d, log₂(d), β/d, n·log₂(q), β·log₂(d), is_dual). Try polynomial features, interaction terms, log₂(q)/n ratio, piecewise features for different β regimes. Try encoding error distribution type (Gaussian vs CBD). All features must be computable at inference time for NIST extrapolation (no realized instance data like error_norm).
2. **Cost Model Architecture:** Go beyond OLS and small MLP. Try gradient-boosted trees (XGBoost/LightGBM), Gaussian processes, symbolic regression for closed-form formulas, polynomial regression, kernel ridge regression, ensembles.
3. **Extrapolation Strategy:** This is the hardest part — from n ≤ 60 to n = 512+. Try log-space modeling, asymptotic-aware models that enforce exponential scaling in β, physics-informed features that bake in Core-SVP structure and fit correction terms. Fit residuals relative to theory rather than raw times.
4. **Per-Attack Modeling:** Primal and dual may have different cost scaling. Try separate models per attack type, or interaction features that let one model adapt. Compare which attack is cheaper in which regime.
5. **Uncertainty Quantification:** Bootstrap training data, fit multiple models, report prediction intervals. Important for the security conclusions.
6. **Theory-Guided Models:** Fit log₂(T) = a·β + b·log₂(d) + correction(n,q,σ). The first term matches Core-SVP structure; the correction is what the agent learns. May extrapolate better than purely data-driven approaches.
7. **δ Prediction (bridges Part A and Part B):** Instead of only predicting runtime, learn to predict root-Hermite factor δ(β, n, q, σ) directly. This would connect the cost model to the security assessment: a better δ predictor means a better empirical β* estimate for NIST schemes. Try multi-task learning (predict both runtime and δ), or fit δ as a secondary model.
8. **Updating stats and notes:** Continuously update NOTES.md and progress.png.
